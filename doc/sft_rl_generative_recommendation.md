# 大模型与生成式推荐中的 SFT 和 RL 阶段

## 1. 一句话理解

SFT（Supervised Fine-Tuning，监督微调）是让模型“学会怎么答、按什么格式答、在业务场景里该模仿什么样的专家行为”。

RL（Reinforcement Learning，强化学习或偏好优化阶段）是让模型“在已经会做任务的基础上，进一步朝业务目标和人类/用户偏好优化”，例如更高 CTR、更长观看时长、更高转化、更少低质推荐、更好的多样性。

放到生成式推荐里：

- SFT 更像行为克隆：给模型用户历史和上下文，让它学习历史中真实发生的点击、播放、购买、停留等正样本推荐结果。
- RL 更像目标对齐：让模型根据 reward model、离线指标、用户偏好或线上反馈，优化最终生成的 item、item list 或推荐解释。

## 2. SFT 阶段怎么做

### 2.1 大模型里的 SFT

大模型预训练后只具备通用语言建模能力，还不一定会按照产品需要完成任务。SFT 通常用人工标注、专家示例、规则构造或高质量日志，把训练样本整理成：

```text
prompt -> ideal response
```

然后用 teacher forcing 做最大似然训练，也就是让模型在给定 prompt 和历史 token 的条件下，尽可能生成目标 response。

典型目标函数是 cross entropy：

```text
maximize log P(response | prompt)
```

SFT 主要解决的问题：

- 任务格式：让模型知道输入是什么、输出应该是什么。
- 领域知识：让模型适配具体业务话术、推荐场景、商品/内容体系。
- 基础能力：让模型先具备可用的指令跟随、推荐生成、排序解释能力。
- 稳定性：让模型输出更可控，减少乱答、格式错误、无关输出。

### 2.2 生成式推荐里的 SFT

生成式推荐可以把推荐任务改造成序列生成任务。输入是用户上下文，输出是 item、item token、item list、推荐理由或下一步行为。

一个常见样本可以长这样：

```text
输入：
用户最近观看：A, B, C
用户画像：北京，偏好直播、本地生活、短视频
场景：美团 App 首页直播入口
候选约束：只推荐可曝光直播间

输出：
D, E, F
```

也可以是：

```text
输入：
用户历史行为序列 + 当前场景 + 时间 + 地理位置

输出：
下一个被点击/观看/购买的 item token
```

生成式推荐 SFT 的数据通常来自：

- 用户点击、播放、购买、停留、收藏、加购等正反馈日志。
- 高质量曝光后转化样本。
- 传统推荐系统产出的高质量排序结果。
- 专家规则或人工精选样本。
- 多目标融合后的 label，例如点击 + 时长 + 转化综合收益较高的 item。

训练方式一般包括：

- next item prediction：根据历史行为预测下一个 item。
- list generation：直接生成一个推荐列表。
- query/user-to-item generation：根据用户上下文生成 item token。
- instruction tuning：让模型按业务指令输出推荐结果、解释或结构化字段。

这里的重点是：SFT 让生成式推荐模型先学会“什么样的用户在什么场景下应该推荐什么”。

## 3. RL 阶段怎么做

### 3.1 大模型里的 RL

大模型里的 RL 通常出现在 SFT 之后。因为 SFT 只是模仿示例，示例不一定等于最优行为。RL 阶段会引入 reward，用来衡量模型输出到底好不好。

典型流程：

1. 先有一个 SFT model，作为初始 policy。
2. 收集偏好数据，例如同一个 prompt 下，response A 比 response B 更好。
3. 训练 reward model，让它预测某个输出的质量分。
4. 用 PPO、REINFORCE、DPO 等方式优化模型，让模型更倾向于高 reward 的输出。
5. 通常会加 KL 约束，让 RL 后模型不要偏离 SFT model 太远，避免 reward hacking 或输出失控。

严格说，DPO 不是传统意义上的 RL，因为它不显式训练 reward model 再跑 PPO；但在很多大模型训练流程里，DPO、IPO、KTO、ORPO 等偏好优化方法承担的是类似 RLHF 的对齐作用。

### 3.2 生成式推荐里的 RL

推荐系统的 RL 可以把推荐过程抽象成：

```text
state: 用户历史、用户画像、场景、时间、地理位置、候选池、上下文
action: 推荐一个 item、生成一个 item list、调整排序结果
reward: 点击、观看时长、转化、GMV、留存、负反馈、多样性等
policy: 生成式推荐模型
```

生成式推荐里的 RL 阶段通常有几类做法。

第一类：基于 reward model 的离线优化。

先训练一个 reward model，预测候选推荐结果的价值：

```text
reward = f(user, context, generated_items)
```

reward 可以来自：

- CTR 预估。
- 观看时长预估。
- CVR/GMV 预估。
- 用户负反馈风险。
- 内容质量分。
- 多样性、新颖性、供给公平性。
- 多目标融合分。

然后让生成式推荐模型生成多个候选 item/list，用 reward model 打分，优化模型更偏向高分结果。

第二类：基于偏好对的优化。

构造推荐结果 A 和 B 的偏好关系：

```text
同一个用户上下文下，推荐列表 A 的综合收益 > 推荐列表 B
```

这个偏好可以来自：

- 线上真实反馈：A 曝光后效果更好。
- 离线 reward model：A 的综合分更高。
- 传统精排/重排系统：A 是更优排序。
- 人工或规则：A 更符合业务约束。

然后用 DPO/IPO 这类 preference optimization 方法训练模型，让它更偏向 A。

第三类：在线 bandit/RL。

在真实流量里探索不同推荐策略，直接用用户反馈更新策略。这类方法收益潜力大，但工程风险也大，需要处理：

- 探索带来的用户体验损失。
- 日志偏差和选择偏差。
- 延迟反馈。
- 多目标冲突。
- 安全边界和业务约束。

实际工业系统通常不会一上来就让大模型完全在线 RL，而是更常见地使用：

- SFT model 生成候选。
- reward model/传统 ranker 做打分和过滤。
- 小流量实验验证。
- 用新日志持续迭代 SFT 数据和 reward model。

## 4. SFT 和 RL 的核心区别

| 维度 | SFT | RL / 偏好优化 |
|---|---|---|
| 本质 | 模仿标准答案 | 优化奖励或偏好 |
| 数据形式 | prompt-response 标注样本 | reward、偏好对、用户反馈、环境反馈 |
| 目标函数 | 最大化目标答案概率 | 最大化期望 reward 或偏好胜率 |
| 训练信号 | “这个答案是什么” | “哪个答案更好”或“这个行为收益多大” |
| 推荐视角 | 学习历史正样本行为 | 优化 CTR、时长、转化、多样性等业务目标 |
| 优点 | 稳定、简单、可控、易落地 | 更贴近真实目标，可优化不可微/长周期收益 |
| 风险 | 只会模仿日志，可能继承旧系统偏差 | 训练不稳定、reward hacking、分布漂移 |
| 典型阶段 | 先做 | SFT 后再做 |

更直观一点：

- SFT 问的是：“在这个场景下，标准答案应该长什么样？”
- RL 问的是：“多个可行答案里，哪个更符合最终目标？”

## 5. 为什么不能只做 SFT

只做 SFT 的问题是，模型会学习日志里的历史行为，但日志不等于最优策略。

推荐日志里常见问题包括：

- 只记录旧系统曝光过的内容，存在 exposure bias。
- 热门 item 被过度学习，长尾 item 学不到。
- 用户点击不一定代表长期满意。
- 高 CTR item 可能低时长、低留存或高负反馈。
- 旧系统的召回/排序偏差会被模型继承。

所以 SFT 可以得到一个“能用的生成式推荐模型”，但如果要对齐更复杂的业务目标，通常还需要 reward model、偏好优化或 RL 阶段。

## 6. 为什么不能直接做 RL

直接从预训练模型做 RL 通常很难，因为模型还不知道推荐任务的基本格式和业务边界。生成式推荐里尤其容易出现：

- 生成不存在的 item。
- 生成不可曝光、不可售卖、不可播放的 item。
- 输出格式不稳定。
- reward 被模型钻空子。
- 训练初期探索空间过大，收敛困难。

因此工业上更稳的路径通常是：

```text
预训练 / 行为序列建模
-> SFT：学会推荐任务和业务格式
-> Reward Model：学习业务收益和偏好
-> RL / Preference Optimization：对齐最终目标
-> 在线小流量实验
-> 用新日志继续迭代
```

## 7. 生成式推荐里的落地流程示例

下面是一个比较工程化的流程。

### 7.1 Item 表达

先决定模型如何表示 item：

- 直接用 item id token。
- 用语义 token，例如类目、tag、embedding 聚类 token。
- 用多级 token，例如 coarse-to-fine 的 item 表达。
- 用文本化描述，例如标题、类目、商家、内容特征。

如果 item 数量极大，直接生成 item id 会有稀疏、冷启动和词表膨胀问题；语义 token 或多级 token 更常见。

### 7.2 SFT 数据构造

从日志里构造：

```text
用户历史行为 + 当前上下文 -> 正反馈 item/list
```

可以按目标筛选高质量样本：

- 点击且观看时长超过阈值。
- 完播、互动、收藏、关注。
- 下单、核销、复购。
- 多目标综合分较高。

也可以把传统推荐系统的高质量结果蒸馏给生成式模型。

### 7.3 SFT 训练

用 cross entropy 训练模型生成目标 item/list。

需要注意：

- 输出必须能映射回真实 item。
- 训练样本要覆盖新老用户、不同场景、不同城市、不同内容供给。
- 可以加入负样本或 hard negative 做对比学习，但纯 SFT 本身通常是正样本 teacher forcing。
- 要避免只学习头部 item，需要做采样平衡或 reweight。

### 7.4 Reward Model 训练

训练一个 reward model 评估推荐结果：

```text
score = RM(user, context, item/list)
```

reward 可以是单目标，也可以是多目标融合：

```text
reward = w1 * CTR + w2 * watch_time + w3 * CVR - w4 * negative_feedback + w5 * diversity
```

推荐场景里 reward 设计是核心难点。设计不好会导致模型追逐短期点击、标题党、重复推荐或低质内容。

### 7.5 RL / 偏好优化

常见做法：

- 让 SFT model 对同一个用户生成多个推荐列表。
- 用 reward model 或真实反馈判断哪个更好。
- 通过 PPO/REINFORCE/DPO 等方式更新生成模型。
- 加 KL 约束，防止模型偏离 SFT model 太远。

训练时通常还要加业务约束：

- 只生成可用 item。
- 去重。
- 类目/内容安全过滤。
- 多样性约束。
- 供给侧约束。
- 新老用户差异化约束。

## 8. 推荐系统工程中的几个坑

### 8.1 Reward 不是越复杂越好

reward 太复杂会难以解释和调试。建议从少量核心目标开始，例如：

```text
点击 + 有效观看时长 + 负反馈惩罚
```

再逐步加入多样性、供给公平性、转化、长期留存等目标。

### 8.2 离线 reward 和线上效果可能不一致

reward model 本身也是模型，会有预估偏差。生成式模型可能学会利用 reward model 的漏洞，离线分高但线上效果不一定好。

因此 RL 阶段必须配合：

- 离线 replay。
- 反事实评估，例如 IPS/DR。
- 小流量 A/B。
- 人工巡检和 badcase 分析。

### 8.3 推荐是 listwise 问题

推荐不是只选一个 item，而是生成一个列表。列表里有位置、重复、多样性、互补性、疲劳度等问题。

所以 RL 阶段最好不要只看单 item reward，还要考虑 list-level reward：

```text
reward(user, context, [item1, item2, ..., itemK])
```

### 8.4 SFT 数据质量决定上限

如果 SFT 数据来自旧系统，模型很容易继承旧系统的偏差。需要做：

- 去重和去噪。
- 头部降采样。
- 长尾和新内容增强。
- 分场景采样。
- 高质量正反馈筛选。
- 对异常点击、误触、低质停留做过滤。

## 9. 面试表达方式

如果面试官问：“生成式推荐里 SFT 和 RL 分别做什么？”

可以这样回答：

> SFT 阶段主要是把推荐任务转成监督学习，让模型根据用户历史、上下文和场景生成目标 item 或推荐列表，本质上是学习历史高质量推荐行为，解决任务格式、领域适配和基础可用性问题。RL 或偏好优化阶段是在 SFT model 基础上引入 reward，例如点击、时长、转化、负反馈、多样性等，通过 reward model、偏好对或在线反馈进一步优化模型，使它不只是模仿日志，而是对齐最终业务目标。两者区别在于，SFT 学的是“标准答案是什么”，RL 学的是“多个答案哪个更好、哪个长期收益更高”。

如果结合你的经历，可以补一句：

> 对生成式推荐来说，我会先用用户行为序列和高质量正反馈做 SFT，让模型具备 next item/list generation 能力；再用精排模型、多目标融合分或线上反馈构造 reward/preference，做 DPO/PPO 类偏好优化，并通过 KL 约束和业务规则保证生成结果不偏离可控范围，最终用小流量 A/B 验证 CTR、时长、转化和负反馈等指标。
